# 暂时移除集中式认证限流设计

日期：2026-07-11

## 背景

当前 Web 与 Admin 的登录、验证码发送和验证码校验共用一套 Redis 限流。该限流按照客户端 IP、标准化账号标识以及 IP/账号组合计数，并在超过阈值时返回 HTTP 429。

生产部署通过 Nginx 反向代理。如果应用服务没有在受控代理边界正确恢复真实客户端地址，`request.client.host` 可能是统一的代理地址，纯 IP 规则会把不同用户合并计数。当前开发阶段也不需要提前确定最终的登录防滥用策略，因此暂时完整拆除这套集中式统计，待产品收尾阶段结合实际部署拓扑重新设计。

## 目标

- 移除 Web 与 Admin 登录的集中式 IP、账号和 IP/账号组合限流。
- 移除验证码发送与校验的集中式 IP、邮箱限流。
- 不再因为集中式统计或限流 Redis 不可用而返回 HTTP 429 或 503。
- 保留验证码自身的业务级安全规则。
- 记录未来重新引入认证防滥用控制时必须解决的设计问题。

## 非目标

- 不改变账号密码校验、未知账号响应或 Argon2 虚拟校验行为。
- 不改变验证码内容、有效期、存储格式或发送流程。
- 不删除验证码业务所需的 Redis。
- 不主动删除已有的 `auth:rate-limit:*` Redis 键；它们按现有 TTL 自动过期。
- 本次不设计替代限流算法，也不增加验证码、设备指纹或验证码挑战。

## 方案

采用完整拆除方案，不保留关闭状态的功能开关或极高阈值。

### 登录

Web 和 Admin 登录路由不再调用集中限流器，也不再为了登录限流注入 Redis 客户端。重复提交登录请求时，接口只返回认证本身的结果：合法凭据返回成功，错误或未知凭据继续返回统一的 401；请求次数不会把结果改变为 429。

本地虚拟管理员逻辑保持不变。它仍只在 `ENVIRONMENT=local` 且 `ENABLE_LOCAL_AUTH_BYPASS=true` 时生效。

### 验证码

注册发码、密码重置发码和验证码校验不再执行 IP 或邮箱累计计数。

以下业务规则继续保留：

- 同一验证码场景的 60 秒重发冷却；
- 冷却期间返回 HTTP 429，并带准确的 `Retry-After`；
- 验证码有效期；
- 单个验证码最多校验 5 次，达到次数后失效。

因此，移除的是集中式统计产生的 429，不是所有 429。

### 代码与配置清理

- 删除集中限流模块及其 Redis Lua 脚本。
- 删除仅供集中限流使用的配置项及正数校验。
- 删除仅供集中限流故障使用的 503 异常。
- 保留 `TooManyRequestsException`，因为验证码重发冷却仍使用它提供 429 和 `Retry-After`。
- 从 `.env.example` 和部署示例中删除集中限流配置。
- 删除或改写只验证集中限流行为的测试和测试夹具。

## 测试设计

- Web 错误登录连续执行超过原阈值后，所有响应仍为统一的 401，不出现 429。
- Admin 错误登录连续执行超过原阈值后，所有响应仍为统一的 401，不出现 429。
- 本地虚拟管理员重复登录继续成功。
- 注册与密码重置发码不再调用集中限流器。
- 验证码重发冷却仍返回 429 和 `Retry-After`。
- 错误验证码的剩余尝试次数和达到上限后的失效行为保持不变。
- 安全配置测试确认已删除的限流配置不再参与启动校验。

## 运行与迁移

本次不需要数据库迁移。代码部署后不再读取旧的限流键，Redis 会根据原有 TTL 自动清理这些键。验证码 Redis 数据和键前缀不变。

## 风险与临时接受

在重新引入防滥用控制前，登录与验证码入口不再具备跨请求的 IP/账号累计保护。现阶段明确接受这一临时风险；验证码重发冷却和单码尝试上限仍提供局部保护。

## 后续恢复清单

最终阶段重新设计认证防滥用控制时，至少需要确认：

1. 可信代理列表以及真实客户端 IP 的恢复边界，禁止直接信任任意 `X-Forwarded-For`。
2. 登录统计是否只记录失败尝试，以及成功登录后如何清零或衰减。
3. Web、Admin、注册发码、密码重置发码和验证码校验分别使用独立阈值。
4. 账号、设备、网络和组合维度的优先级，以及办公网络、NAT 和共享出口的误伤策略。
5. Redis 故障时各环境的 fail-open、fail-closed 和降级行为。
6. 中文错误文案、`Retry-After`、客户端倒计时和可访问性。
7. 告警、指标、审计日志和阈值调优所需的数据最小化原则。
8. 自动化测试覆盖代理拓扑、并发原子性、边界次数和跨实例行为。
