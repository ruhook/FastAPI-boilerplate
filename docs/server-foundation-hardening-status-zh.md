# Server 底层加固状态

更新时间：2026-07-11

## 已完成

- 生产配置 fail-closed：密钥、CORS、本地免登录/引导开关、SMTP、OSS 与邮件加密配置均有启动校验。
- 本地免登录保留为 `ENVIRONMENT=local` + `ENABLE_LOCAL_AUTH_BYPASS=true` 的显式虚拟超级管理员，不写数据库。
- 请求日志集中脱敏，SMTP 授权码加密存储且不再通过查询接口回传。
- access token 包含账号、portal、版本与 token 类型；普通 refresh session 可轮换、撤销并检测重放。
- 密码散列已直接替换为 Argon2id；bcrypt 依赖和旧散列兼容入口均已移除，未知、禁用和错误密码路径执行等量 dummy verify。
- 注册/重置验证码发送不再暴露账号是否存在，SMTP 异常不回传，验证码 HMAC 与 SMTP 密钥解耦。
- 邮件事件通过事务 outbox 提交；消费者使用有界队列和并发 worker，成功后才 ACK，公平处理新消息与 stale pending，并在达到投递上限后原子写入 DLQ 再 ACK。
- 招聘状态修改增加行锁、确定性锁顺序、版本号与 409 冲突响应。
- 候选人投递通过 MySQL 生成列和唯一索引保证“同一账号、同一岗位只有一条有效申请”，并在同一事务内用 SQL 原子更新申请人数；并发重复投递和不同用户同时投递均有真实 MySQL 回归测试。
- 资产上传按扩展名和文件字节共同识别，只持久化服务端 MIME；HTML/SVG/XML/JS、签名不匹配和未知二进制会被拒绝。只有 JPEG/PNG/GIF/WebP 可 inline，其余文件强制 attachment，并统一添加 `nosniff` 和 CSP。
- 资产只序列化鉴权 API URL；上传有大小/分块限制、本地路径防越界、数据库失败时补偿删除对象；批量 ZIP 有文件数/总字节上限并使用临时 spool 流式返回。
- Admin 附件和 Admin 业务数据继续对所有已认证 Admin 开放，这是明确产品规则；候选人附件仍按本人所有权/业务引用校验。
- 普通管理员改为角色显式授权；迁移为旧账号创建兼容全权角色；管理员列表角色读取消除 N+1。
- 候选人申请 summary 改为流式扫描、在线聚合，只保留当前页 ORM 对象和合同记录。
- 自动筛选文本 `eq` 与数值 `eq` 已分流处理，文本规则不再被数值解析提前判失败。
- 不保留 bcrypt、客户端 MIME、旧表单 `attachment/single_select`、旧事件 pending 或废弃配置别名等兼容分支；开发数据采用重置、重建或重新上传。
- 底层加固回归使用专用 `hr_server_codex_test` + 本机 Redis；全仓 Ruff 与核心 mypy 门禁已通过，Alembic 已确认只有 `20260710_000045` 一个 head。
- CI 使用一次性 MySQL/Redis 并要求显式的测试库清理许可；验证码重发冷却、有效期和单码尝试上限以及取消集中限流后的路由行为由独立测试覆盖。

## 仍需继续处理

- 登录、验证码发送与校验的集中式 IP/账号限流已在当前开发阶段暂时移除；验证码重发冷却、有效期和单码尝试上限仍保留。最终恢复要求见 [暂时移除集中式认证限流设计](superpowers/specs/2026-07-11-disable-central-auth-rate-limit-design.md)。
- 历史超大对象的单文件读取仍需真正的对象流式传输；当前分类仍不是杀毒、内容净化或沙箱服务。
- 资产软删除后的保留期、物理删除任务和双向孤儿对账命令尚未实现。
- 自动应付、推荐奖励等依赖动态计算和全量 summary 的列表仍会扫描较多匹配数据，需要后续通过快照/物化字段或等价 SQL 聚合优化。
- `job_progress/service.py` 仍然过大，命令、查询、展示和合同/邮件协调尚未完成模块拆分。
- mypy 目前只是明确的核心模块门禁，不是全仓业务模块类型通过；历史业务模块类型债务仍需分批清理。
- 候选人申请展示查询仍有一条 SQLAlchemy 笛卡尔积告警，需要单独检查关联条件和执行计划；FastCRUD `create()` 还有一个未来主版本返回类型变更告警。
- DLQ 当前只有 Redis 状态和日志，没有管理、重放、告警与积压看板。
- 审计日志扩面、邮件大附件/大 payload 容量、部署文档校正和全仓依赖门禁仍属于后续 P2 工作。

## 当前判断

本轮其他约定的 P0/P1 底层风险已完成回归和静态门禁验证，但集中式登录与验证码防滥用保护仍明确处于暂缓状态，不能认为所有认证安全边界已经闭合。除这项已接受的临时风险外，其余问题主要是可计划偿还的 P2 生命周期、查询性能、模块拆分、运维能力和类型覆盖债务。
